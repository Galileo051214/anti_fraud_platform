## 1. 项目总体架构

### 技术栈

| 层级 | 技术选型 | 版本 |
|------|---------|------|
| 前端框架 | Vue 3 + TypeScript | Vue 3.4+ |
| 构建工具 | Vite | 5.x |
| 状态管理 | Pinia | — |
| UI 组件库 | Element Plus | — |
| CSS 预处理器 | SCSS (Less 部分页面) | — |
| 后端框架 | Spring Boot 3 | 3.2.x |
| ORM | MyBatis-Plus | — |
| 数据库 | MySQL 8 | — |
| 缓存 | Redis | — |
| AI 接口 | DeepSeek API | — |
| 安全 | Spring Security + JWT | — |
| 文档 | Knife4j (Swagger) | — |

### 为什么要这样选型

- **Spring Boot 3 + Vue 3**：前后端分离，团队中全栈开发者可以独立并行开发前后端，互不阻塞
- **TypeScript**：类型安全在前端复杂数据流（嵌套评论、推荐结果、闯关状态）中极大减少了运行时错误
- **MyBatis-Plus**：相比 JPA 更灵活，复杂 SQL（如排行榜排名计算、威尔逊得分排序）可以直接写原生 SQL
- **Pinia**：比 Vuex 更轻量、TypeScript 支持更好，适合管理用户登录态和闯关进度等跨组件状态
- **JWT 无状态认证**：不需要服务端存储登录会话，便于后续水平扩展

## 2. 用户管理模块

### 功能说明

提供完整的用户生命周期管理，包括注册、登录、个人信息维护、密码修改、管理员对用户的管理操作。用户体系分为两种角色：

- **student（学生）**：平台主要内容消费者，参与学习、闯关、交流
- **admin（管理员）**：管理后台内容、用户、查看统计数据

### 设计亮点

#### 2.1 安全密码存储
```java
// SecurityConfig 中配置 BCrypt 编码器
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```
**为什么这样做**：BCrypt 自带随机盐值，相同密码加密结果不同，防止彩虹表攻击。相比 MD5/SHA 这种快哈希，BCrypt 的设计就是慢哈希，大幅提高暴力破解成本。

#### 2.2 双重 Token 校验
```java
// JwtAuthenticationFilter 中校验 Token 的同时检查黑名单
if (redisCacheUtil.isTokenBlacklisted(token)) {
    throw new BusinessException(401, "Token已被列入黑名单");
}
```
**为什么这样做**：JWT 签发后无法主动失效，通过 Redis 黑名单机制实现了"登出即失效"的安全需求。Token 本身仍然是无状态的，黑名单只在登出时写入一份数据。

#### 2.3 数据脱敏
```java
// DesensitizationUtils 对手机号、邮箱等敏感信息脱敏
public static String maskPhone(String phone) {
    return phone.replaceAll("(\\d{3})\\d{4}(\\d{4})", "$1****$2");
}
```
**为什么这样做**：在 API 返回用户列表时自动脱敏，避免用户手机号、邮箱等隐私信息在前端或日志中泄露。

## 3. 反诈资讯模块

### 功能说明

资讯模块是平台的核心内容板块之一，负责发布和管理反电信网络诈骗相关的新闻、预警通知和政策法规。支持分类管理、置顶、必读机制和浏览统计。

### 设计亮点

#### 3.1 三级资讯类型

```
news（普通资讯）
├── warning（预警通知）
└── policy（政策法规）
```
**为什么这样做**：不同类型的资讯有不同的紧急程度和展示权重。预警通知需要突出显示，政策法规需要结构化归档，普通资讯则更灵活。三级分类在数据库中用 `news_type` 枚举字段区分，避免建多个表带来的查询复杂度。

#### 3.2 必读机制
```java
// NewsServiceImpl 中获取必读资讯
public List<News> getRequiredNews(Long userId) {
    // 1. 查出所有 isMandatory = 1 的资讯
    // 2. 排除用户已阅读过的
    // 3. 用户首次登录或管理员新发布时触发
}
```
**为什么这样做**：学校场景下，管理员有时需要确保某些重要通知（如新型诈骗预警）被所有学生阅读。必读机制 + 前端弹窗提醒的组合，比纯靠学生主动点击更可靠。

#### 3.3 异步浏览统计
```java
// 记录浏览 + 增加浏览量
@Transactional
public void recordBrowse(Long newsId, Long userId) {
    newsBrowseLogMapper.insert(log);
    newsMapper.incrementViewCount(newsId);
}
```
**为什么这样做**：虽然每次浏览都写一次数据库看起来"重"，但这是为了后续做推荐系统采集数据。`news_browse_log` 表中的浏览时间和停留时长，是推荐算法计算用户兴趣标签的重要输入。

| `GET /api/news/required` | 必读资讯列表 | 登录 |

## 4. 诈骗案例库模块

### 功能说明

案例库是平台的核心知识资产，收录了各类电信网络诈骗的真实案例改编内容。每个案例包含诈骗类型标签、剧本脚本（决策树）、目标用户定向、威尔逊置信度评分等。

### 设计亮点

#### 4.1 剧本决策树（Scripts）
```
案例 "刷单返利诈骗"
├── 节点1: 诈骗分子发布刷单广告
│   ├── 用户选择「尝试一单」 → 节点2
│   └── 用户选择「不予理会」 → 安全结局
├── 节点2: 小额返利获取信任
│   ├── 用户选择「继续刷单」 → 节点3
│   └── 用户选择「提现退场」 → 节点4
└── ...
```
**为什么这样做**：传统的图文案例阅读完就结束了。剧本决策树让用户"沉浸式"体验诈骗过程，在关键节点做出选择，看到不同的结果。这种互动式学习的记忆效果远好于被动阅读。FSM（有限状态机）模式在 `ScenarioServiceImpl` 中实现，状态流转清晰可维护。

#### 4.2 威尔逊得分排序
```java
// RecommendationAlgorithmUtil 中实现
public static double wilsonScore(long likes, long total, double z) {
    double p = (double) likes / total;
    double z2 = z * z;
    return (p + z2 / (2 * total) - z * Math.sqrt(
        (p * (1 - p) + z2 / (4 * total)) / total
    )) / (1 + z2 / total);
}
```
**为什么这样做**：简单按点赞率排序对小样本不友好（1/1=100% 的案例会排在第 100/200=50% 之前）。威尔逊区间考虑了样本量，点赞数越少置信度越低，得分自动向下修正，确保高排名案例是真正的"高口碑"。

#### 4.3 定向推送
```java
// FraudCase 实体中的目标字段
private String targetGrades;    // ["大一","大二"]
private String targetMajors;    // ["计算机系","金融系"]
```
**为什么这样做**：不同年级、不同专业的学生面临的主要诈骗类型不同。大一新生更需要"刷单诈骗"的案例，金融系学生更容易遇到"投资理财诈骗"。定向推送让每个学生看到最相关的内容。

#### 4.4 标签体系
案例通过 `case_tag` 和 `case_tag_relation` 实现多对多标签关联。标签包含名称、分类、颜色，同时服务于推荐算法中的标签匹配和筛选。

### 核心 API

| 端点 | 说明 | 权限 |
|------|------|------|
| `GET /api/case/page` | 案例分页 | 公开 |
| `GET /api/case/{id}` | 案例详情 | 公开 |
| `GET /api/case/hot` | 热门案例 | 公开 |
| `GET /api/case/wilson` | 威尔逊得分排序 | 公开 |
| `POST /api/case` | 创建案例 | 管理员 |

## 5. 闯关学习模块

### 功能说明

闯关学习是平台最具互动性的核心模块，包含三种子模式：**答题闯关**、**情景模拟** 和 **Agent AI 挑战**。学生通过"闯关地图"逐步解锁关卡，每通过一关获得积分和经验。

### 设计亮点

#### 5.1 三合一闯关模式

| 模式 | 类型值 | 交互方式 | 核心机制 |
|------|--------|---------|---------|
| 答题闯关 | `quiz` | 单选/多选/判断 | 自动评分，即时反馈 |
| 情景模拟 | `scenario` | FSM 决策树 | 上下剧情关联选择，多结局 |
| Agent 挑战 | `agent_scenario` | AI 自然语言对话 | 与 AI 扮演的诈骗分子对话 |

**为什么这样做**：三种模式从"基础知识点掌握"到"情景判断力训练"再到"实战应对能力"，层层递进。答题闯关适合知识测试，情景模拟培养判断力，Agent 挑战训练真实对话应对能力。

#### 5.2 FSM 状态机引擎（情景模拟）

[ScenarioServiceImpl.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/service/impl/ScenarioServiceImpl.java) 实现了基于有限状态机的情景模拟引擎：

```java
// 核心数据结构：ScenarioProgress
// - currentNode: 当前所在剧情节点
// - decisionHistory: 用户所有决策路径
// - status: in_progress / completed / failed

// 每次用户提交决策：
// 1. 校验当前节点是否存在该选项
// 2. 记录决策到 history
// 3. 根据选项指向跳转到下一个节点
// 4. 如果到达 ending 节点，计算最终评分
```
**为什么这样做**：FSM 比流程图引擎轻量得多，不需要额外的可视化编排工具。剧本数据以 JSON 存储在 `challenge.scripts` 字段中，结构清晰，管理员可以直接在后端编辑 JSON 维护剧本。

#### 5.3 Agent AI 挑战（流式对话）

[AgentChallengeServiceImpl.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/service/impl/AgentChallengeServiceImpl.java) 使用 DeepSeek API 实现 AI 角色扮演：

```java
// AntiFraudPromptTemplate 中构造的角色扮演提示词
public static String buildAgentChallengePrompt(String scenarioConfig) {
    return """
        你正在扮演一个电信网络诈骗分子，你需要设法骗取用户的信任...
        当前角色设定：%s
        你的目标是让用户上钩，但用户可能识破你的骗局。
        请根据用户之前的回应调整你的话术。
        """.formatted(scenarioConfig);
}
```
**为什么这样做**：
- **流式响应（SSE）**：AI 生成文字需要时间，流式输出让用户看到打字效果，体验比等待完整响应好得多
- **评分报告**：会话结束后，AI 根据用户对话表现自动生成评分报告（`scoringReport`），分析用户哪些回复应对正确、哪些存在风险
- **每日奖励限制**：`agent_challenge_daily_reward` 表控制每日练习次数，防止过度依赖 AI 而忽略基础知识学习

#### 5.4 闯关地图可视化

前端闯关主页以"地图"形式展示所有关卡，每个关卡卡片显示：
- 关卡编号和名称
- 难度星级（1-5）
- 通关状态（未开始/进行中/已通过/未通过）
- 最佳得分

**为什么这样做**：游戏化的地图展示比列表更有"闯关"的沉浸感，激发学生的通关欲望。类似 Duolingo 的学习路径设计被证明能有效提升用户留存率。

| 端点 | 说明 | 模式 |
|------|------|------|
| `GET /api/challenge/list` | 闯关列表 | 通用 |
| `POST /api/challenge/submit` | 提交答题 | quiz |
| `POST /api/scenario/start/{id}` | 开始情景模拟 | scenario |
| `POST /api/scenario/decision` | 提交决策 | scenario |

## 6. 社区互动模块

### 功能说明

论坛社区为学生提供了经验交流、求助问答和讨论互动的空间。包含帖子发布与管理、嵌套评论系统、点赞互动、积分激励等功能。

### 设计亮点

#### 6.1 三级帖子类型

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| `experience` | 经验分享 | 分享学习心得、被骗经历警示 |
| `question` | 求助问答 | 遇到疑似诈骗寻求帮助 |
| `discussion` | 讨论交流 | 讨论热点诈骗手法 |

**为什么这样做**：分类发帖便于后续的内容聚合和推荐。例如"求助问答"类帖子通常有较高的时效性和互动需求，可以优先推送给活跃用户。

#### 6.2 安全 HTML 渲染 + 自定义颜色

[HtmlSanitizer.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/common/HtmlSanitizer.java) 实现了差异化的 HTML 净化策略：

**帖子正文**：保留完整富文本标签（h1-h6, p, ul, ol, table, blockquote 等）+ 安全的 CSS 颜色属性
```java
SAFELIST = Safelist.relaxed()
    .addTags("h1", "h2", "h3", "h4", "h5", "h6", "pre", "code", ...)
    .addAttributes(":all", "class", "style")
    .addProtocols("a", "href", "http", "https", "mailto")
```

**评论内容**：仅保留内联标签（b, i, u, em, strong, span, s, a）+ CSS 颜色样式
```java
COMMENT_SAFELIST = Safelist.none()  // 从零开始构建
    .addTags("b", "i", "u", "em", "strong", "span", "s", "a")
    .addAttributes(":all", "style")
```

**CSS 安全过滤**：只允许颜色相关的 CSS 属性，禁止 `url()`、`javascript:` 等危险值：
```java
// 白名单属性
ALLOWED_CSS_PROPERTIES = {"color", "background", "background-color",
                          "font-weight", "font-style", "text-decoration",
                          "text-align", "font-size"}

// 危险值检测
isSafeCssValue("url(http://evil.com)")    // false
isSafeCssValue("javascript:alert(1)")     // false
isSafeCssValue("red")                     // true
```

**为什么这样做**：

- **XSS 防护**：直接使用 `v-html` 渲染用户内容必须做后端净化，JSoup 的白名单模式比黑名单过滤更安全——白名单明确"允许什么"，黑名单总有遗漏的可能
- **差异化策略**：帖子正文使用富文本编辑器、样式需求更多；评论使用纯文本输入框、仅支持少量内联标签，降低攻击面
- **CSS 属性白名单**：允许用户修改文字颜色和背景色，但禁止 `background-image`（可能用于追踪像素）、`width/height`（可能破坏布局）等

#### 6.3 嵌套评论系统

评论支持无限层级嵌套回复（通过 `parentId` 字段），前端展开/收起控制显示：

```java
// CommentServiceImpl 中的树形查询
private void collectCommentTreeIds(Long commentId, List<Comment> postComments,
                                    Set<Long> commentIds) {
    if (!commentIds.add(commentId)) return;  // 防止环形引用
    for (Comment comment : safeList(postComments)) {
        if (commentId.equals(comment.getParentId())) {
            collectCommentTreeIds(comment.getId(), postComments, commentIds);
        }
    }
}
```
**为什么这样做**：一次查询出某个帖子的所有评论，在内存中以递归方式构建树结构，避免 N+1 查询问题。虽然评论量大时对内存有压力，但考虑到学生社区的使用规模，一次查询几百条评论完全在可接受范围内。

#### 6.4 发帖积分激励

```java
// CommentServiceImpl 中创建评论后
scoreService.addScore(userId, 1, "评论互动");
leaderboardService.updateScore(userId, 1, "daily/weekly/all");
```
**为什么这样做**：每发一个帖子或写一条评论都给予少量积分，正向激励用户参与社区互动。积分虽少，但积少成多，学生会为了提升排行榜排名而更积极发帖。

| `POST /api/forum/post` | 创建帖子 | 登录 |

## 7. 智能客服模块

### 功能说明

基于 DeepSeek 大语言模型的反诈 AI 问答助手。学生可以随时向 AI 咨询反诈相关问题，AI 会结合 RAG（检索增强生成）技术从平台案例库、资讯库中检索相关内容作为回答依据。

### 设计亮点

#### 7.1 Prompt 工程（系统提示词）

[AntiFraudPromptTemplate.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/util/AntiFraudPromptTemplate.java) 中包含了精心设计的系统提示词：

```java
public static String buildSystemPrompt() {
    return """
        你是一个专业的反电信网络诈骗AI助手。你的核心职责是：
        1. 识别和解释各类诈骗手法
        2. 提供防范建议和应对策略
        3. 分析可疑情况，判断是否为诈骗
        4. 引导用户在日常使用中提高防骗意识

        回答风格要求：
        - 专业准确，但用语通俗易懂
        - 以真实案例为基础说明问题
        - 对于不确定的问题，明确告知用户
        - 遇到疑似正在发生的诈骗，优先建议报警

        参考信息：%s
        """.formatted(context);
}
```
**为什么这样做**：精心设计的 Prompt 让 AI 的回答风格和内容范围可控，不会偏离"反诈助手"的定位。`context` 参数注入 RAG 检索结果，让 AI 的回答有据可依，减少"幻觉"。

#### 7.2 RAG 检索增强
```java
// AntiFraudAgentServiceImpl 中
public String askQuestion(String question, Long userId) {
    // 1. 从案例库和资讯库检索相关内容
    List<String> relevantDocs = retrieveRelevantDocs(question);
    // 2. 注入到 Prompt 的 context 中
    String prompt = buildSystemPrompt(relevantDocs);
    // 3. 调用 DeepSeek API
    String answer = deepSeekClient.chat(prompt, question);
    // 4. 记录问答日志（含来源引用）
    saveConversation(userId, question, answer, relevantDocs);
    return answer;
}
```
**为什么这样做**：纯靠大模型的知识库回答反诈问题不可靠（大模型不知道最新诈骗手法，也可能"幻觉"出虚假案例）。RAG 让 AI 的回答建立在实际案例和资讯的基础上，回答末尾还能标注参考来源。

#### 7.3 风险等级标记

每条问答记录 AI 会自动评估风险等级：
- `low`：正常咨询
- `medium`：用户描述的情况有可疑点
- `high`：用户可能正在遭遇诈骗

**为什么这样做**：高风险问答可以推送给管理员关注，必要时主动联系学生提供帮助。这是"AI 辅助 + 人工介入"的防诈闭环。

#### 7.4 SSE 流式响应
```java
// ChatController 中使用 SseEmitter
@PostMapping("/ask-stream")
public SseEmitter askStream(@RequestBody ChatRequest request) {
    SseEmitter emitter = new SseEmitter(0L); // 不超时
    // 异步调用 DeepSeek 流式 API
    deepSeekClient.chatStream(request.getQuestion(), chunk -> {
        emitter.send(SseEmitter.event().data(chunk));

    return emitter;
}
```
**为什么这样做**：大模型生成完整回答需要 3-10 秒，如果等全部生成再返回，用户会对着空白界面干等。SSE 流式返回让用户看到 AI "逐字输出"的效果，交互体验显著提升。

## 8. 推荐系统模块

### 功能说明

推荐系统是平台"千人千面"的核心支撑，根据用户画像（年级、专业、知识水平）、行为数据（浏览记录、闯关成绩、标签偏好）和生命周期阶段，向用户推荐最相关的内容（案例、资讯、闯关）。

### 设计亮点

#### 8.1 四维混合推荐算法

[RecommendationAlgorithmUtil.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/util/RecommendationAlgorithmUtil.java) 实现了四种推荐策略的加权融合：

```
推荐得分 = α × 内容匹配 + β × 协同过滤 + γ × 生命周期 + δ × 关联规则

各维度权重根据用户当前生命周期动态调整：
- newbie（新手期）：α=0.5, γ=0.3, β=0.1, δ=0.1  → 侧重基础内容
- growing（成长期）：α=0.3, β=0.3, γ=0.2, δ=0.2  → 均衡推荐
- mature（成熟期）：α=0.2, β=0.4, γ=0.1, δ=0.3  → 侧重个性化和探索
```

**为什么这样做**：

| 维度 | 算法 | 解决的问题 |
|------|------|-----------|
| 内容匹配 | TF-IDF + 标签匹配 | 新用户没有行为数据时，根据年级/专业推荐相关内容 |
| 协同过滤 | 皮尔逊相关系数 | 找到"相似的人"，推荐他们喜欢的内容 |
| 生命周期 | 阶段规则 | 新手看基础、老手看深度 |
| 关联规则 | SPM 序列模式挖掘 | "看过刷单诈骗的人，还看了投资诈骗" |

**单一推荐算法都有局限性，四维混合通过权重组合弥补各自的短板。**

#### 8.2 用户生命周期管理

[ProfileServiceImpl.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/service/impl/ProfileServiceImpl.java) 根据用户行为自动判定生命周期阶段：

| 阶段 | 判定条件 | 策略 |
|------|---------|------|
| `newbie`（新手） | 注册 < 7 天 或 闯关通过 < 3 | 推荐基础资讯和入门案例 |
| `growing`（成长） | 通过学习 >= 3 关，知识水平提升 | 均衡推荐各类内容 |
| `mature`（成熟） | 高活跃 + 高知识水平 | 推荐深度案例和新颖内容 |

**为什么这样做**：给大一新生和研三学长推荐同样的内容显然不合理。生命周期自动判定的机制让推荐系统"自适应"用户成长过程，不需要维护同学手动调整偏好设置。

#### 8.3 用户-标签行为矩阵

每个用户对每个标签的"行为得分"随时间衰减：

```sql
-- user_behavior_matrix 表
id | user_id | tag_id | behavior_score
-- behavior_score = 浏览+1, 点赞+3, 闯关通过+5, ...

-- 每日定时任务衰减分数，让长期不活跃的兴趣标签权重降低
UPDATE user_behavior_matrix
SET behavior_score = behavior_score * 0.95
WHERE ...;
```
**为什么这样做**：用户的兴趣是变化的。上个月对"刷单诈骗"感兴趣，不代表这个月还想看。时间衰减机制让推荐结果始终反映用户近期的兴趣偏好。

#### 8.4 用户相似度计算

```java
// 皮尔逊相关系数实现
public static double pearsonSimilarity(Map<Long, Double> vectorA,
                                        Map<Long, Double> vectorB) {
    // 计算两个用户在共同标签上的行为相关性
    // 1.0 = 完全正相关（兴趣一致）
    // -1.0 = 完全负相关（兴趣相反）
    // 0.0 = 无关联
}
```
**为什么这样做**：协同过滤依赖"相似的用户"。皮尔逊相关系数考虑了用户打分尺度的差异（A 用户所有行为分都偏高，B 用户都偏低，但偏好趋势一致），比余弦相似度更适合行为数据。

#### 8.5 SPM 序列模式挖掘

```java
// 从行为日志中挖掘: "先浏览A → 再浏览B"的序列模式
// 生成关联规则: {浏览"刷单诈骗"} → {浏览"投资诈骗"}, confidence=0.75
```
**为什么这样做**：有些诈骗类型之间存在先后关联。学生先学到"刷单诈骗"后，往往需要了解更深层的"投资诈骗"。SPM 挖掘这种序列关系，让推荐具有"递进学习"的逻辑。

## 9. 积分成就模块

### 功能说明

积分和成就是平台的两大激励引擎。积分用于量化用户的学习投入，成就用于标记用户的关键里程碑。两者共同构建"学有所得"的正反馈循环。

### 设计亮点

#### 9.1 积分体系

| 行为 | 积分 | 上限 |
|------|------|------|
| 闯关通过 | +10~30（视难度） | 不限 |
| 发布帖子 | +5 | 不限 |
| 发表评论 | +1 | 不限 |
| 每日首次登录 | +3 | 每日 1 次 |
| 完善个人信息 | +5 | 一次性 |
| 解锁成就 | +50~200 | 每次成就 |

**为什么这样做**：积分设计采用了"高频低额 + 低频高额"的组合。评论和每日登录是高频行为，积分低但可以积少成多；闯关通过和成就解锁是低频高价值行为，积分高，激励用户持续学习。

#### 9.2 等级体系

等级基于 `totalScore` 自动晋升：

```java
// ScoreServiceImpl 中根据总积分计算等级
public int calculateLevel(int totalScore) {
    if (totalScore < 100) return 1;
    if (totalScore < 300) return 2;
    // ... 每 200 分为一级（等级越高，升级越慢）
    return 10 + (totalScore - 1500) / 500;
}
```
**为什么这样做**：等级所需积分呈指数增长，前期升级快让新用户获得成就感，后期升级慢保持挑战性。

#### 9.3 成就系统

| 成就代码 | 名称 | 解锁条件 |
|---------|------|---------|
| `FIRST_LOGIN` | 初次登录 | 首次登录平台 |
| `FIRST_CHALLENGE` | 初出茅庐 | 完成第一个闯关 |
| `CHALLENGE_MASTER` | 闯关达人 | 通过全部闯关 |
| `FIRST_POST` | 畅所欲言 | 发布第一个帖子 |
| `KNOWLEDGE_STAR` | 知识新星 | 累积获得 500 积分 |
| `ANTI_FRAUD_EXPERT` | 反诈专家 | 累积获得 2000 积分 |

**为什么这样做**：成就是比积分更高阶的激励手段。积分是"量"的积累，成就是"质"的突破。徽章墙展示在前端个人中心，社交展示价值（"我有你没有"）本身就是一种激励。

## 11. 缓存系统模块

### 功能说明

基于 Redis 的缓存系统，负责加速热点数据访问、降低数据库查询压力、缓存排行榜计算结果。

### 设计亮点

#### 11.1 分层缓存架构

```
┌─────────────┐     ┌──────────────┐     ┌──────────┐
│  前端浏览器  │────▶│ 应用层缓存   │────▶│  MySQL   │
│  (无缓存)    │     │  (Redis)     │     │          │
└─────────────┘     └──────────────┘     └──────────┘
                         │
                    ┌────┴────┐
                    │ 内存缓存  │
                    │(CopyOnWrite│
                    │ArrayList) │
                    └─────────┘
```
**为什么这样做**：Redis 缓存"热数据"（如首页轮播、排行榜），内存缓存"紧急预警"这种需要实时推送给所有用户的数据。双层缓存互补，Redis 持久化可靠，内存缓存零网络开销。

#### 11.2 定时刷新 + 缓存预热

[CacheServiceImpl.java](file:///d:/Project/anti_fraud_platform/backend/src/main/java/com/anti/service/impl/CacheServiceImpl.java) 中的定时任务：

```java

public void refreshHotNewsCache() {
    // 刷新 10/20/50 三种粒度的热点资讯缓存
}

@Scheduled(fixedRate = 60000)  // 每1分钟
public void refreshLeaderboardCache() {
    // 排行榜需要更实时，1分钟刷新一次
}
```
**为什么这样做**：定时刷新 + 缓存预热确保热门数据"常驻"缓存，用户访问时几乎全部命中缓存，极少触发数据库查询。

#### 11.3 缓存一致性策略

```java
// 写入操作触发缓存失效
@Transactional
public News updateNews(Long id, UpdateNewsRequest request) {
    News news = super.updateById(id, request);
    redisCacheUtil.deleteNewsDetail(id);     // 失效详情缓存
    redisCacheUtil.deleteHotNews();          // 失效热点列表
    return news;
}
```
**为什么这样做**：Cache-Aside 模式，先更新数据库，再删除缓存。下次读取时发现缓存未命中，从数据库重新加载并写入缓存。这个模式在大多数场景下都能保证数据最终一致性。
